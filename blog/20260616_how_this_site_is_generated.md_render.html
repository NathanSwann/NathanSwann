How This Site Is Generated

← AOC Day 3 Index

It’s no secret that I despise just about anything to do with browsers: HTML, CSS, JS, 100s of random dependencies and random browser quirks. In a world where a lot of things are React mono-apps, the thought of making this blog put me into a spiral of finding the right solution. Ultimately the entire job of this blog is to display text in a readable, friendly manner. For that, things should be simple.

Given I don’t want to just be writing raw HTML, the most common path (to my knowledge) is static site generators like Jekyll or Hugo, but they bring with them a lot of baggage. I want to rebuild my entire site in less than a second, I don’t overly care about fancy features, I just want something that works the way I want it and no more.

Something Done Right

With all that said, I decided that if I wanted a static generator that works my way I would have to make it. So let’s go over the requirements of this blog:

  1. Super minimal dependencies - The generator should never break when I update something or switch computers, it should depend on the barest of minimal requirements.
  2. Render Markdown - I want to be able to write Markdown rather than HTML.
  3. Navigation - You should be able to click between posts, a simple TOC and index should be enough.
  4. Simple Styling - Nothing flashy, I’m not trying to develop the next dopamine-addicting app, just a blog.

The One Dependency

The hardest part of this is rendering Markdown, thankfully a tool exists that can turn a Markdown page into HTML: pandoc. I actually use this everywhere in my day to day and it’s a super stable and well supported program so no issues with dependency hell. I could stop here with a

pandoc --no-highlight blog.md

but like I said I want a bit more.

The Actual Build Script

Since I’m using pandoc, a simple bash script that builds my site via pandoc is basically all I need. The entire generation for this site is around 60 lines of bash and a couple of HTML templates with CSS styling. So how does it work? Well let’s break it down into its actual steps:

  1. Collect blog post files
  2. Render each blog
  3. Render the index

Step 1 is literally just a find blog -name "*.md" command put into an array.

Rendering A Singular Blog

For simplicity each blog is rendered into the same template, the shortened layout of which is this:

Template layout

<!DOCTYPE html>
<html lang="en">
<head>
    <title>{{title}}</title>
</head>
<body>
<h1 class="post-title">{{title}}</h1>
<nav class="post-nav">
<span class="post-nav-prev">{{prev_nav}}</span>
<span class="post-nav-index"><a href="/blog/index.html">Index</a></span>
<span class="post-nav-next">{{next_nav}}</span>
</nav>
<div id="content">
<!----CONTENT_BEGIN---->
</div>
</body>
</html>

The names are fairly self-explanatory. All the script needs to do is take a Markdown file and figure out what its title is, the posts before and after it, and the actual content. I store the title in the actual filename (this post’s filename is 20260616_how_this_site_is_generated) and run sed to get a nicely formatted title by removing the date and fixing the casing. Similarly, since we collect everything beforehand, the prev and next navs can just take the previous and next post from the array. Now for the content, you might think that this is some insane code… it’s not.

This one is actually kind of interesting though, for no other reason than it lets me show off some nice sed tricks. You could pipe the output of pandoc into a file and then splice that file in using sed, or we can tell sed to use pandoc! To do this I’m using the e mode of sed rather than the usual g or d. e tells sed to insert the output of the following command before the matched line, for example:

echo -e "test\ntest2" | sed '/test2/e whoami'
test
nathan
test2

So in order to render we can do:

sed /<!----CONTENT_BEGIN---->/e pandoc path_to_markdown

The full blog generation is this, in case you are curious:

Full generation code

title_from_path() {
    basename "$1" | sed -e "s/[^_]*_\(.*\)\.md/\1/" | sed "s/_/ /g" | sed -e "s/\b\(.\)/\u\1/g"
}

mapfile -t ALL_POSTS < <(find blog -name "*.md" | sort)
N=${#ALL_POSTS[@]}

generate_blog() {
    local idx=$1
    local p=${ALL_POSTS[$idx]}
    local blog_name
    blog_name=$(title_from_path "$p")

    local prev_nav=""
    local next_nav=""

    if [ "$idx" -gt 0 ]; then
        local prev_p=${ALL_POSTS[$((idx-1))]}
        local prev_title prev_url
        prev_title=$(title_from_path "$prev_p")
        prev_url="/blog/$(basename "$prev_p")_render.html"
        prev_nav="<a href=\"${prev_url}\">← ${prev_title}</a>"
    fi

    if [ "$((idx+1))" -lt "$N" ]; then
        local next_p=${ALL_POSTS[$((idx+1))]}
        local next_title next_url
        next_title=$(title_from_path "$next_p")
        next_url="/blog/$(basename "$next_p")_render.html"
        next_nav="<a href=\"${next_url}\">${next_title} →</a>"
    fi

    sed "s|{{title}}|${blog_name}|g" templates/blog.html \
        | sed "s|{{prev_nav}}|${prev_nav}|g" \
        | sed "s|{{next_nav}}|${next_nav}|g" \
        | sed "/<!----CONTENT_BEGIN---->/ e pandoc --no-highlight ${p}" \
        > "${p}_render.html"

    echo "Built: $p"
}

for i in "${!ALL_POSTS[@]}"; do
    generate_blog "$i"
done

The Index and TOC

The index page works the same way. I split a template at a marker, emit a <ul> of posts in reverse chronological order, stitch it back together. The table of contents is the one thing that runs client-side: a small JS snippet queries #content for headings on page load and builds the list from whatever pandoc generated.

Index Generation Code

generate_index() {
    {
        sed -n '/<!--INDEX_CONTENT-->/{ q }; p' templates/blog_index.html
        echo "<ul class=\"index-list\">"
        for ((i=N-1; i>=0; i--)); do
            local p=${ALL_POSTS[$i]}
            local title url date
            title=$(title_from_path "$p")
            url="/blog/$(basename "$p")_render.html"
            date=$(basename "$p" | grep -oE '^[0-9]{8}' | sed 's/\([0-9]\{4\}\)\([0-9]\{2\}\)\([0-9]\{2\}\)/\1-\2-\3/')
            echo "<li><span class=\"index-date\">${date}</span> <a href=\"${url}\">${title}</a></li>"
        done
        echo "</ul>"
        sed -n '/<!--INDEX_CONTENT-->/,$p' templates/blog_index.html | tail -n +2
    } > blog/index.html
    echo "Built: blog/index.html"
}

What I Didn’t Do

Since this is tailored to my needs there are a lot of things I haven’t done. For example, hot reloading/auto-rebuilding, I just run my build script with watch, no need for something fancier. I don’t have any fancy plugin systems or anything since again this is a simple blog, it just needs to work for me.

I guess my main takeaway here is that you don’t actually need a big fancy static generator or big React apps most of the time. Instead a bunch of bash utils can get you 90% of the way. This system is still performant (actually more so than some of the bigger generators) and covers all of my use cases. Not only that but it’s trivial to add new stuff to, since there isn’t some monolith that has made a bunch of assumptions for me. Overall I think this site is staying generated like this for a while.
